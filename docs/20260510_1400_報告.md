# Git ブランチ運用ルール

## 1. 概要

本プロジェクトでは `main` ブランチを使用せず、以下のブランチ構成で開発を行います。実際のビルド対象は **`release/forBuild`** とします。

### ブランチ構成図

* **`release/forBuild`** (ビルド用・デフォルトブランチ)
* **`release/v-X.X.X/candidate`** (リリース候補)
* **`feature/v-X.X.X/candidate`** (機能統合)
* **`develop/v-X.X.X/*`** (個別機能開発)
* **`* /fix/*`** (各階層の修正)
* **`other/*`** (その他)



---

## 2. ブランチ詳細

### 2.1 release ブランチ

#### `release/v-X.X.X/candidate`

* **用途**: 完成したコードの管理、動作確認・デバッグ・最終調整済みのリリース候補管理。
* **ルール**:
* 更新時は新しいバージョンブランチを作成する（例: `release/v-1.0.0/candidate`）。
* 完成後、`release/forBuild` へマージする。


* **注意事項**:
* **Push禁止**。必ずPull Request（PR）を使用。
* 複数人によるレビューと動作確認を必須とする。



#### `release/forBuild`

* **用途**: 実際のゲームビルド実行用。

### 2.2 feature ブランチ

#### `feature/v-X.X.X/candidate`

* **用途**: 機能統合および結合テスト用。`develop` ブランチで完成した機能をまとめる。
* **ルール**: `release` ブランチと同じバージョンを使用する（例: `feature/v-1.2.3/candidate`）。
* **注意事項**:
* **Push禁止**。必ずPRを使用。
* 結合テスト完了後に `release` へマージする。



### 2.3 develop ブランチ

#### `develop/v-X.X.X/<feature-name>`

* **用途**: 個別機能の開発。
* **ルール**:
* ブランチ名にバージョンを含める。
* 機能単位で作成し、内容がわかる名称にする（例: `develop/v-1.0.0/player-move`）。


* **注意事項**: 単体テスト完了後に `feature` へマージする。

### 2.4 fix ブランチ

* **用途**: 不具合修正（develop用、feature用、release用緊急修正）。
* **命名規則**:
* `develop/v-X.X.X/fix/<fix-name>`
* `feature/v-X.X.X/fix/<fix-name>`
* `release/v-X.X.X/fix/<fix-name>`


* **例**: `release/v-1.0.0/fix/crash-title`

### 2.5 other ブランチ

#### `other/*`

* **用途**: BGM追加、イラスト追加、一時検証、小規模修正など。
* **例**: `other/bgm-add`, `other/temp-ui`
* **注意事項**: 追跡が必要な重要な開発には使用しないこと。

---

## 3. 開発フロー

1. **feature作成**: `feature/v-0.1.0/candidate` を作成。
2. **develop作成**: `develop/v-0.1.0/player-move` 等を作成して開発。
3. **develop → feature**: 単体テスト完了後、PR経由でマージ。
4. **feature → release**: 結合テスト完了後、PR経由でマージ。
5. **release → release/forBuild**: 最終確認後、PR経由でマージし、ビルド実施。

---

## 4. GitHub 設定

### 4.1 基本設定

* **Default Branch**: `release/forBuild` に変更。
* **main ブランチ**: 保護のみ行い、実運用（Push/Merge）は禁止。

### 4.2 Branch Protection (release/forBuild)

* [x] Pull Request 必須
* [x] Approve 必須
* [x] Force Push 禁止 / Delete 禁止
* [x] Status Check 必須

### 4.3 マージルール

| マージ元 → 先 | 推奨形式 |
| --- | --- |
| `develop` → `feature` | Squash merge 可 |
| `feature` → `release` | Merge commit 推奨 |
| `release` → `release/forBuild` | **Merge commit 必須** |

---

## 5. 自動化 (GitHub Actions)

* **develop**: Lint, Unit Test
* **feature**: Integration Test
* **release**: Export Build, Artifact Upload
* **release/forBuild**: Steam / itch.io Upload, Release Zip 作成
